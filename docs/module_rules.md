# Правила Модулей

## Правила описания модулей
Каждый модуль имеет описание в файле docs\modules\[название модуля]_module.md
Описание модуля обязательно состоит только из разделов:
- Назначение - название и назначение краткое,
- Публичный интерфейс - список всех api и их параметров с кратким описанием каждого параметра и функции,
- Публичные контракты - список всех dto с полями с кратким описанимем каждого параметра и назначения dto
- Список используемых модулей
- Алгоритм работы и его особенности - краткое, структурированное описание. Длина описания максимум 1000 символов (если не влезает, то делаем саммери чтобы влезло).

## Публичная Поверхность

Публичный api модуля описывается в его описании, в модуле не должно содержаться api и контрактов не описанных в архитектуре, а так же кода их реализующего (если такие находятся, то УДАЛЯЮТСЯ)
`api.py` — единственный публичный фасад модуля.
`contracts.py` содержит только DTO, значения Enum/Literal, объявления Protocol/ABC и публичные
исключения. В нем запрещены ввод-вывод, запуск подпроцессов, импорт MLflow и импорт приватных
файлов реализации.
Приватные файлы реализации называются `_*.py`.
Внешний код импортирует только из `<module>.api` и `<module>.contracts`.

## Границы
Один модуль не импортирует `_*.py` файлы другого модуля.
Публичный интерфейс каждого модуля должен совпадать с `docs/modules/<module>_module.md` и
`tests/test_public_contracts.py`.
Любое добавление в публичный интерфейс требует обновления документации модуля и тестов публичного
интерфейса.
Устаревшие обертки запрещены.
Новые модули верхнего уровня запрещены без соответствующего документа в `docs/modules/`.
